## 1. 目的とコンセプト

### 1.1 ビジョン
> 「頭の中のノイズを減らし、今日やるべきことだけを目の前に置く」

自身のADHD的特性（タスクの先送り、取りこぼし、何から手をつければいいか分からなくなる状態）をシステムで物理的に防ぎ、AIのサポートで「とりあえず着手する」ハードルを極限まで下げる、自分専用の実用型タスク管理ツール。

### 1.2 解決したい課題
- **計画の困難さ**: 何から手をつければいいか分からず、着手できない。漠然とした目標を実行可能な単位に落とせない。
- **忘却**: リマインドがないと存在自体を忘れる。習慣は特に「今日はいいか」が積み重なって消滅する。

### 1.3 達成したいこと（優先順位順）
1. 今日やらなければならないことを落とさない
2. タスクをとりあえず記録してこぼさない（キャプチャの摩擦を最小化）
3. 習慣を風化させずに可能な限り実践する
4. 未来の完了に向かって計画的にことを進めていく

### 1.4 設計原則
- **記録の摩擦は極小に、振り分けの摩擦はUIで吸収する**（Inboxに全部投げ込める→後で仕分け）
- **Project / ToDo / Habit の3区分で管理する**。「やりたいこと」のような主観的な区別はシステム上の型としては設けず、必要ならタグで表現する（2.2節参照）
- 習慣トラッキングは「失敗しても責めない」設計にする（ストリークが切れても罰を強調しない）
- AIは代行者ではなく「補助輪」。最終判断とタスクの所有権は常にユーザーにある

## 2. 用語定義（ドメインモデル）

### 2.1 セグメント定義
柔軟な親子関係をベースとした3セグメントで管理する。

- **Project（大きな塊・長期目標）**
  - 完了までに複数の手順が必要な「大きなタスク」。最終的な締め切り（期日）を設定可能
  - 直下にToDoを子タスクとして持つ
- **ToDo（単発・実行タスク）**
  - 単発の行動。完了条件が明確なもの。日時指定・期日の設定が可能
  - **親子関係**: ToDo自身にも、さらに細かな子タスク（孫タスク以降も含め無制限）を作成できる（＝「大きなタスクをBaby Stepに分解する」は、単にToDoの子ToDoを作ることと同義）
  - **Memoの扱い**: Markdownメモは「期限なしのToDo」の一種として扱う。独立した別エンティティは用意しない
- **Habit（習慣・ルーチンカタログ）** ※別テーブル
  - 繰り返し発生する行動のマスターデータ
  - 「デイリープランナー（Habitピッカー）」機能によって、当日分のToDoとして実体化（インスタンス生成）させて運用する

### 2.2 分類・タグ
- カテゴリのような固定の単一選択分類は採用せず、**タグ（自由・複数付与可）** に一本化する
- 「開発」「Amazon事業」「私生活」のような大分類も、「やりたいこと」のような意欲ベースの区別も、すべて自由タグで表現する（例: `#開発`, `#want`）

### 2.3 その他の用語
| 用語 | 説明 |
|---|---|
| **Inbox Item** | 仕分け前の生の入力。タイトルのみ、または自然文（例：「歯医者にいく 6/17 15:00」）で日時情報を含みうる |
| **Reminder** | ToDo/Habitに紐づく通知トリガー（時刻・繰り返し・スヌーズ設定を持つ） |
| **Recurrence Rule** | 繰り返し規則。ToDo/Habit双方に設定可（例: 毎日/毎週火木/月末/n日おき/「週3回以上」など柔軟な頻度指定） |

### 2.4 状態遷移
```
Inbox → (仕分け) → Project or ToDo or Habit
Project/ToDo: 未着手 → 進行中 → 完了 / 先送り(reschedule) / 破棄
Habit: 状態遷移ではなく「実践ログ」の蓄積（continuous）。デイリープランナーが当日分のToDoを生成
```

## 3. 機能仕様

### 3.1 Inbox / クイックキャプチャ
- グローバルにどこからでも開ける入力欄（キーボードショートカット / モバイルは常駐+ホーム画面ウィジェット❓）
- **爆速Inbox（Smart Input）**: テキスト一行で投げ込め、記号で属性を自動付与する
  - `#タグ名` : タグ指定
  - `!Project名` : 親Project指定
  - `今日` `明日` `10/24` : スケジュール設定（自然文からの日時抽出。誤解釈を前提に必ずプレビュー→確定のステップを挟む）
- 仕分けするまでInboxに滞留。仕分けを促すバッジ（未仕分け件数）を常時表示
- **トリアージUI**: Inboxリストから1件ずつ（またはスワイプで）Project/ToDo/Habitに振り分け。振り分け時に最低限の追加情報（期日、親Project所属など）を聞く

### 3.2 Today ビュー
- 「今日やらなければいけないこと」に特化した画面。ToDo（期日=今日 or 期限超過）と今日実践すべきHabitのみを表示
- 未仕分けInboxはここには出さない（認知負荷を分離するのが目的）
- 完了・先送り（明日以降にリスケジュール）をワンタップ/ワンスワイプで

### 3.3 デイリープランナー（Habitピッカー）
- Habitマスターから、当日分のToDoインスタンスを生成する専用画面
- 頻度指定は柔軟に: 「毎日」「週n回（曜日固定 or 自由）」「n日に1回」
- ストリーク表示はするが、途切れても否定的な表現をしない（❓トーン設計は要すり合わせ：グラフのみ/励ましメッセージあり、など）
- 「今日まだやっていない」を穏やかに知らせるリマインダー（強制感の少ない文言）

### 3.4 ToDoの細分化（子ToDo・孫ToDo）
- 任意のProject/ToDoに、無制限に子ToDoを追加できる（Baby Step化はこの子ToDo作成と同一の操作）
- **AI分解機能**: 重たいToDo/Projectに対し、AIが5〜15分程度の極小な子ToDoを自動提案・生成。ユーザーは提案を編集・並べ替え・削除して確定
- 分解後は「次にやるべき1個」だけを強調表示するモード（選択麻痺の回避）

### 3.5 AIによる能動的サポート
- **スケジューリング支援**: 既存の予定（カレンダー）とタスクの残量から、「いつやるか」の提案を出す
- **ヒアリング型着手支援**: ユーザー側からの相談だけでなく、AIから「これやろうぜ」と声をかけ、対話形式で状況を聞きながら子ToDoを一緒に組み立てるモード
- **介入トリガー（初期案）**: 作成/最終更新から **1週間以上放置** されたProject/ToDoを対象に、AIから提案を出す。頻度・トーンなど詳細は別途詰める（❓）
- **多言語タスク名ジェネレーター（将来機能・MVP対象外）**: 日本語タスク名をAIが英語・韓国語に翻訳・併記提案する語学学習支援機能

### 3.6 リマインダー
- ToDo/Habit単位で複数リマインダーを設定可能（例: 前日20時 + 当日朝 + 期限1時間前）
- スヌーズ（時間指定 or 定型「1時間後」「明日」）
- Android Web Pushで通知（対応端末はAndroidのみのためiOS制約は考慮不要）

### 3.7 繰り返しタスク
- ToDo/Habit共通の繰り返しエンジン（cron的ルール + 「n日おき」「第n曜日」等の柔軟指定）
- 親タスク完了時に次回分を自動生成する（繰り返しタスクの完了はその回のインスタンスのみ完了扱いにし、次回分は自動生成）

### 3.8 Markdownメモ
- Project/ToDoに紐づく自由記述メモ（Markdown記法対応、プレビュー切り替え）
- 単体の雑記帳的メモも「期限なしのToDo」として作成すればよく、別エンティティは不要（2.1節参照）

### 3.9 外部連携
- **Googleカレンダー同期**: MVPスコープ。一方向（Zendo→Calendarへ、日時確定ToDoをイベントとして反映）から開始し、将来的に双方向同期を検討する
- **Googleスプレッドシート連携**: MVP対象外。将来検討（用途案: 完了ログやHabit記録の自動エクスポート）

### 3.10 モバイル体験（PWA、Android想定）
- スワイプ操作: 完了/先送り/削除をカード型リストのスワイプに割り当て
- Android Web Push通知
- オフライン時もInboxへのキャプチャ・Todayビュー閲覧は可能（Service Worker + IndexedDBキャッシュ、後でSupabaseに同期）

## 4. 画面構成（案）

1. **Inbox** — 未仕分けキャプチャ一覧 + クイック追加
2. **Today** — 今日のToDo + 今日のHabit（デイリープランナー含む）
3. **Planning / Calendar** — 週表示、Project/ToDoをドラッグして日付に配置
4. **Habits** — 習慣一覧、継続率、実践ログ
5. **Projects** — Project一覧、各配下ToDoの進捗
6. **Notes** — Markdownメモ（期限なしToDo）一覧（横断検索）
7. **Settings** — リマインダーのデフォルト、Google連携、AIの介入設定

## 5. データモデル

詳細設計は **[database-design.md](database-design.md)** に確定版として分離した（そちらを正とする）。概要:

- `items` テーブル1つに Inbox / Project / ToDo を `kind` 列で同居（トリアージ＝kind更新、ToDo⇄Project昇格降格も自由）
- 習慣は `habits`（マスター）＋ デイリープランナーが生成する「habit_id付きToDo行」（実践ログは導出、専用ログテーブルなし）
- `is_memo` は専用列で確定（タグ方式はタイプミス事故のため不採用。旧❓解消）
- 繰り返しは「完了時に次回1件だけ生成、過去分は積み上げない」セマンティクス（ADHD配慮の中核設計。詳細は database-design.md 4章）
- リマインダーはルール（相対/絶対）と解決済み発火時刻の2層で保持
- CalendarSyncLink はMVP対象外のため未作成（将来テーブル追加のみで対応可能な設計）

## 6. 技術スタック（決定事項）

| レイヤ | 選定 |
|---|---|
| フレームワーク | **Next.js（App Router）+ TypeScript**。フロントエンドとバックエンド（API Routes）を同一プロジェクトで管理し、特定のBaaSのサーバーレス機能に依存しない構成にする |
| スタイリング/UI | **Tailwind CSS + shadcn/ui**（shadcn/uiはコードを直接リポジトリにコピーする方式のため、中身がブラックボックス化しない） |
| フロントエンド配信形態 | PWA一本化（ホーム画面追加、Web Push、Service Workerでオフライン対応）。対応端末はAndroidのみを想定 |
| データベース | **Supabase Postgres** のみ利用（Auth基盤・Realtimeは現時点で不使用、Edge Functionsも使わない）。関係モデルの複雑さ（Project/ToDo/Habit/Reminderの関連）からPostgresを選定 |
| DBアクセス経路 | **クライアントはSupabaseと直接通信せず、すべてAPI Routes経由**（v0.4確定）。理由: (a) 繰り返し完了時の次回生成等をサーバーでアトミックに実行 (b) 将来のDB乗り換え時にクライアント無変更 (c) DB接続キーをブラウザに露出させない。詳細は database-design.md 2章 |
| バックエンドロジック | Next.jsの **API Routes**（Vercelにデプロイ）が担う。Claude API呼び出し・Google OAuth処理などの秘密情報を扱う処理はすべてここに集約し、特定BaaSの無料プラン仕様変更に依存しないようにする |
| AI連携 | Claude API をNext.js API Routes経由で呼び出し、APIキーはサーバー側（Vercelの環境変数）のみで保持 |
| ホスティング | Vercel（Next.js標準のデプロイ先） |
| 認証 | なし。専用URL（推測困難なパスまたはBasic認証程度の軽い保護）で運用 |
| 外部連携 | Google Calendar API（一方向同期、MVPスコープ、将来双方向検討）。Google Sheets連携はMVP対象外（将来検討） |

### 6.1 Supabase 無料プランの運用上の注意
- 用途をPostgres（DBのみ）に絞ったため、Edge Functionsの無料枠等は考慮不要。懸念点は主にDB本体の可用性のみ
- Free Planは **7日間APIアクセスがないとプロジェクトが自動pause** される。毎日使う前提なので通常運用では発生しない想定
- 長期不在（旅行・体調不良等）でpauseした場合、ダッシュボードから手動Unpauseで復帰可能（データは保持される）
- pauseがさらに長期間（90日超）続くとプロジェクト削除のリスクがあるため、必要であれば「毎日1回叩く軽量ヘルスチェックcron（Vercel Cron等の無料枠）」でpause自体を回避する対策を後付けできる。MVPでは対策なしで様子を見て、実際に長期不在の見込みが立った時点で対応する

❓ **認証なし方針とGoogle連携の整合性**: Googleカレンダー同期にはOAuthが必須（Googleアカウントへの権限付与）。「ログイン認証なし」はアプリ自体へのログイン画面を指すが、Google連携の初回にはOAuth同意フローが別途発生する点は認識合わせしたい。

## 7. 非機能要件

- 個人利用前提のため多重ユーザー・権限管理は不要
- レスポンス速度: Inboxへの追加は体感即時（楽観的UI更新、バックグラウンドでSupabaseに同期）
- プライバシー: 個人のタスク内容がAI APIに送信される点をユーザー自身が許容している前提（第三者データではないため大きな懸念は低いが、送信範囲は最小化）

## 10. MVPスコープ

### 10.1 MVPに含める（土台）
- Inbox / 爆速キャプチャ（`#タグ` `!Project` `今日/明日/日付`）
- トリアージUI（Project/ToDo/Habitへの振り分け）
- Today ビュー
- デイリープランナー（Habit→当日ToDo生成、頻度は「毎日」「曜日固定週次」のみ）
- リマインダー（複数設定・スヌーズ）
- 繰り返しタスクエンジン（ToDoの繰り返し。親完了で次回自動生成）
- ToDoの子ToDo化（手動のみ、AI分解なし）
- Markdownメモ（期限なしToDoとして）
- PWA基本機能（Android Web Push、Service Workerによるオフラインキャプチャ）
- 認証なし・専用URL運用

### 10.2 MVP対象外（後回し）
- AI機能全般（ベイビーステップ自動分解、能動的声かけ、スケジューリング支援）
- Googleカレンダー同期
- 習慣頻度ルールの柔軟化（週n回自由選択、n日おき等）
- 多言語タスク名ジェネレーター
- Googleスプレッドシート連携

理由: Inbox/Today/Habit/リマインダーという「毎日使う土台」を先に完成させ、実運用しながら土台の使用感を固めた上でAIレイヤーを乗せる方が、手戻りが少ないため。

## 11. 次のステップ

仕様検討はここで一区切り。次は実装フェーズに移行する。
