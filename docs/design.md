# Zendo デザインシステム

UI実装時は必ず本ドキュメントに従う。モック議論の最終合意（2026-07-15）を集約したもの。

## 0. 原則 — 機能美

1. **色は意味にだけ使う**。装飾のための色は存在しない。「色が付いている＝何か意味がある」を全画面で成立させる
2. チェックボックス・入力欄・ナビ・罫線など**普遍的なUIは無彩色**（形が機能を説明している）
3. 色だけで意味を伝えない。**チップには必ずテキスト**（色覚多様性配慮）
4. 装飾ラベル（"TODAY"等の英字見出し）・カードの箱と影は使わない。区切りは罫線、区画は生成り面
5. 白地ベース。ステータス色が何色乗っても飽和しない
6. **例外: 習慣の継続表示（10章）だけは派手を許可**——モチベーション装置として、大きさ・太さ・面積を解放する。ただし色の意味論（常磐=達成・紅=警告）は維持する。この例外を他の画面に波及させないこと

## 3. 画面仕様（実装済み分）

### Today
- 見出し: 「M月D日（曜）」＋右に「のこり n件」(nibi)。装飾ラベルなし
- リスト: 期限超過＋今日のToDo（習慣インスタンス含む）
- 「今日の習慣」= kinari地の角丸区画。候補行＋「＋ 追加」(mikan文字ボタン)
- データ: `GET /api/today` → `{ date, todos, habitCandidates, done }`（doneは当日JSTに完了したもの）

### Inbox
- 見出し: 「Inbox」＋「未仕分け n件」
- 仕分けは**「今日」「明日」の2択のみ**（無彩色枠ボタン）＝期日を設定するだけの操作（8章の統合後はkind変更を伴わない）。Project化は詳細モーダルの「⋯→Projectに変換」
- 入力欄はTodayと同一（スマホ=FAB / PC=リスト末尾常設）

## 7. タスク詳細モーダル（実装済み）

全リストの行タップで開く編集画面。モックは https://claude.ai/code/artifact/d4c79c75-5379-4f7b-90a2-364f89184700 参照。

### 7.1 基本構造

- PC=中央カード（幅430px・最大高でスクロール）/ スマホ=全画面モーダル。構成は完全同一
- 閉じる: PC=✕・Esc・外側タップ / スマホ=「←」＋Androidの戻るジェスチャ
- **項目ごとの自動保存**（保存/キャンセルボタンなし）。上部に「変更は自動保存されます」を小さく常設
- 縦の並び: タイトル → 期日 → 繰り返し → リマインダー → プロジェクト → タグ → **メモ → 子ToDo**（この順）
- **右上「⋯」メニュー**に低頻度操作を集約: Projectに変換 / 破棄（繰り返しを終了） / 削除（beni）。破棄・削除は確認ダイアログ1回。フッターの操作行は置かない

### 7.2 フィールド仕様

| 項目 | 挙動 |
|---|---|
| タイトル | タップでその場編集。空にしたら元値に戻す（空タイトルは保存しない） |
| 期日 | 行タップで直下にピッカー展開（7.3）。✕クリア=期日なしタスクに戻る（8章の統合によりInboxビューへ自然に現れる）。変更時は相対リマインダー自動再計算（API実装済み） |
| 繰り返し | 行タップで直下にセグメント展開: なし/毎日/毎週(曜日複数選択)/毎月(日指定)/n日おき(n＋起点schedule/completion)。期日なしでは設定不可（先に期日設定を促す） |
| リマインダー | 複数可、各行✕で削除。「＋追加」プリセット: **当日の朝8:35 / 1時間前 / 30分前 / 10分前 / 日時指定**。「n分前」系は期日に時刻がない場合グレーアウト（理由をツールチップ） |
| プロジェクト | タップで既存プロジェクトの選択リスト（検索付き）or「なし」。=parent_id の付け替え |
| タグ | チップ表示＋「＋」でテキスト入力追加。チップタップで削除 |
| メモ | プレーンテキスト複数行（Markdownプレビューは将来機能） |
| 子ToDo | チェックリスト表示（チェック=完了トグル・tokiwa塗り）。行タップで**その子の詳細にモーダル内容が切替、「←」で親に戻る**（モーダルは重ねない）。「＋子ToDoを追加」でインライン入力 |

### 7.3 期日ピッカー

- インライン展開（モーダル内モーダルは作らない）。構成: 日付input＋時刻input（上段）→ カレンダー（下段）。ショートカットチップは**不採用**
- **日付input**: 手入力可・ゆるいパース（`7/19` `2026/07/19` `0719` を受理）。不正入力は赤枠＋元値維持。**カレンダーと双方向同期（必須要件）**
- **時刻input**: ネイティブ`type="time"`。任意項目、✕で削除=終日。時刻を消すと「n分前」系リマインダーは自動削除（警告表示）
- **カレンダー**: 自作。月曜始まり・‹›で月移動・今日=枠線・選択日=mikan塗り・過去日は淡色だが選択可。日付タップで即設定しピッカーは開いたまま（続けて時刻を触れる）
- 確定ボタンなし（タップ即保存）。ピッカー外タップ or 期日行の再タップで畳む

### 7.4 kind別の表示差分

- **8章のデータモデル統合により、Inbox項目用の仕分け行は廃止**。「ToDoにする」は消滅（期日設定がその役割を果たす）、「Projectにする」は「⋯→Projectに変換」に統合。モーダルは全アイテム同一形
- **kind=project のみ**: 繰り返し・リマインダーの2行を非表示（**期日行は表示**——プロジェクト自体の期日として設定可能。ただし一覧の行に表示するのは子由来の「次の期日」のみ、9章）。子ToDoセクションがプロジェクトの中身一覧を兼ねる

### 7.5 着手中（doing）の廃止（実装済み）

- status は `todo / done / dropped` の3値。`20260717000001_unify_data_model.sql` でenumを再作成し、既存の `doing` 行は `todo` に変換済み（kind/statusはCHECKではなくPostgres enumだった点に注意）

## 8. データモデルの統合 — inbox状態の廃止（実装済み）

- kind を **task(`todo`) / `project` の2種に統合**。「inbox」という状態・遷移は概念ごと廃止する
- **Inboxは状態ではなくビュー**。表示条件: `kind='todo' AND due_date IS NULL AND parent_id IS NULL AND is_memo=false AND status='todo'`
- 期日を設定→Inboxビューから消える / 期日を✕クリア→Inboxビューに現れる。**特別な遷移ルールは存在しない**（すべて検索条件の帰結）。プロジェクトの子は親を持つためInboxに出ない（期限なしでも可）。メモは`is_memo`でNotesへ分離
- マイグレーション `20260717000001_unify_data_model.sql` で適用済み（7.5のdoing廃止・10.1の頻度刷新と同一バッチ）
- Inboxビューのクエリ定数は `src/lib/client.ts` の `INBOX_QUERY`（一覧とバッジで共用し条件のズレを防ぐ）
- `captured_raw` は従来通り（キャプチャ由来タスクの生文字列保持に使う。Smart Inputが確定時に保存）

## 10. Habits画面（実装済み）

モックは https://claude.ai/code/artifact/d4c79c75-5379-4f7b-90a2-364f89184700 参照。この画面は「習慣の設定と継続の確認」に純化（実行系はTodayのプランナーに一本化）。継続表示は0章6の例外規定により**意図的に派手**。

### 10.1 頻度モデルの刷新（DB変更）

- `frequency_rule` の語彙: `{type:"daily"}` / `{type:"every_n_days", n}` / `{type:"times_per_week", n}` の3種。**曜日指定(weekly/weekdays)は廃止**——暦固定ルーチンは繰り返しタスク（recurrence_rule）で作る棲み分け
- マイグレーション: 既存 `{type:"weekly", weekdays:[...]}` は `{type:"times_per_week", n:weekdays.length}` へ変換
- 「週」は**月曜はじまり**（カレンダーと統一、JST）
- Todayプランナー候補の出方（**完了ログ依存になる点に注意**。実装は frequency.ts の書き換え＋/api/today）:
  - daily: 毎日出る（当日インスタンス未生成なら）
  - every_n_days: 最後に完了した日からn日経過で出る。完了実績ゼロなら毎日出る
  - times_per_week: 今週(月〜日)の完了数がn未満なら毎日出る。達成したら出ない

### 10.2 継続の指標（すべて完了ログから導出、保存列は増やさない）

- **連続記録（ストリーク）**: daily=連続日 / every_n_days=連続サイクル / times_per_week=連続達成週。当日・当週がまだ未達成でも期間が終わるまで途切れ扱いにしない
- **おやすみ救済**（全頻度。2026-07-17に times_per_week も対象化）: 該当期間を1回逃してもストリーク継続、表示が「おやすみ中」に変わる。次の該当期間に達成すれば復帰（おやすみ分は数に入れない）。**2連続で逃した時点で途切れ**——途切れの通知・表現は一切しない（チップが黙って消えるだけ）。判定単位: daily=日 / every_n_days=前回完了から2n日以内 / times_per_week=週（前週が未達成でも1回まで許容、2週連続未達成で途切れ）
- **今週の進捗**: daily=「今週 5/7」/ times_per_week=「今週 1/3」(達成で「今週 3/3 達成！」常磐) / every_n_days=「次は今日/明日/M月D日」
- **直近4週の達成率**: 編集モーダル内のみ表示（一覧には出さない）

### 10.3 一覧（カードUI・派手ゾーン）

- カード構成: 名前＋頻度ラベル / **週バー**（daily=7マス・times_per_week=nマス、実施分を常磐で塗る。every_n_daysはバーなし）/ **進捗テキスト（太字）** / 右上に**ストリーク大数字**（26px級・常磐。「12 日連続」「4 週連続」「8 回連続」）
- **達成カード**（今週n/n）は面ごと tokiwa-soft に染める
- **おやすみ中**: ストリーク数字を無彩色化＋「(おやすみ中)」＋**紅の警告バナー**「あと1日で連続8回が失われます — 今日やれば守れる」（損失回避＋救済可能性をセットで）
- **Todayタスク化ボタン**（カード右下・3状態）: 「＋ 今日やる」（mikan塗りピル・既存instantiate APIで当日インスタンス生成）→「Todayに追加済み」（鈍色テキスト）→「✓ 今日は完了」（常磐テキスト）。当日未生成なら非該当日・達成済み週でも押せる（余分にやるのは自由）
- 一時停止中の習慣は下部の折りたたみ「一時停止中 n件 ▸」へ退避
- カードタップ（ボタン以外）→ 編集モーダル

### 10.4 編集モーダル

- タスク詳細（7章）と同じ骨格・自動保存。冒頭に**統計ヒーロー**（ストリーク大数字＋週バー＋進捗＋直近4週達成率、tokiwa-soft面）
- 頻度: セグメント［毎日／n日に1回／週n回］＋数値ステッパー（−/＋）
- 既定リマインダー1つ（プリセットは7.2と同じ語彙。生成される当日ToDoに付与）/ タグ / 一時停止トグル（「再開するまで候補に出ません」）/ メモ
- 「⋯」メニュー: 削除のみ（過去の実践ログ=完了済みインスタンスは残す）
- 追加: スマホFAB / PC常設入力欄。名前だけで作成（既定=毎日）、頻度はモーダルで変更

## 11. Smart Input（実装済み）

モックは https://claude.ai/code/artifact/d4c79c75-5379-4f7b-90a2-364f89184700 参照。

### 11.1 基本フロー

- 適用範囲: **Today / Inbox / Notes のタスク入力**（インライン＋FABシート。Notesは13.2で追加）。Projects/Habitsの追加入力は名前だけの単純入力のまま
- 入力中、解釈結果を**リアルタイムでチップ列プレビュー**（PC=入力欄の直下、FABシート=入力欄の上=キーボードに隠れない位置）。表示: タイトル文字列＋[期日チップ(asagi)]＋[#タグチップ]＋[!プロジェクトチップ]
- Enter確定でチップ内容をそのまま適用。**チップの✕=その解釈だけ取り消し**、該当文字列はタイトルに残す（同一入力中は再解釈しない）
- 確定時に元の生文字列を `captured_raw` に保存（既存列。将来の再パース用）

### 11.2 語彙

| 入力 | 解釈 |
|---|---|
| `今日` `明日` `明後日` | 相対日付 |
| `M/D` `M月D日` | 絶対日付。過去日なら来年に繰り上げ |
| `◯曜日` `◯曜` | **必ず未来**の直近のその曜日（今日は含まない。今日なら「今日」と打つ運用） |
| `HH:MM` `H時` `H時半` | 時刻。**日付指定がなければ当日**とみなす |
| `#タグ名` | タグ（複数可、空白区切り） |
| `!名前` | プロジェクト紐付け（→parent_id）。下記11.3 |
| 残りの文字列 | タイトル（空白整理） |

- **日付・時刻は最初の1つだけ解釈**。2つ目以降の日付表現は文字のまま（「明日 8/16の準備」→期日=明日、タイトル「8/16の準備」）
- 繰り返し語彙（毎日/毎週等）はMVP対象外（詳細モーダルで設定）。将来追加枠

### 11.3 !プロジェクトのサジェスト

- `!`入力でサジェストポップオーバーを開き、続く文字で既存プロジェクト名を部分一致絞り込み（↑↓/タップで選択）
- リスト末尾に**「＋ 新規プロジェクト『◯◯』を作成」ボタン**（明示的な作成のみ。**暗黙作成はしない**）
- 選択せずに確定した場合: 一意に一致すれば紐付け、一致なし/複数一致なら文字のまま残す

### 11.4 挙動の既定値と実装ノート

- Today入力: 日付トークンなし→従来通り今日。明示があれば優先（金曜のタスクとして作成され、プレビューで行き先が見える）
- Inbox入力: 日付トークンなし→期日なし（Inboxビューに残る）。あれば期日つき（Todayなどへ）
- パーサは純粋関数（`src/lib/smart-input.ts` 想定）として分離し、ユニットテストを付けることを推奨（テスト追加はCLAUDE.md規約に従い実装時に承認を取る）
- 入力文中の認識部分の着色ハイライトは後回し（チッププレビューが機能要件、着色は演出）。**実装はプレーンinputに縛らない**——contenteditable等の採用も実装者の裁量

## 12. この先の予定（設計確定 2026-07-21）

### 12.1 解決する問題

`期日 > 今日` の最上位タスクは、どのビューの条件にも合致せず**どこからも見えない**状態だった（Inbox=期日なし / Today=期日<=今日 / Projects=kind=project）。副作用として:

- Inboxで「明日」に仕分けた瞬間、そのタスクが当日中は不可視になる
- **繰り返しタスクは完了した瞬間に消える**（次回インスタンスは必ず未来日付のため）。繰り返し設定が生きているか確認する手段がなかった

### 12.2 置き場所と構成

**Inbox画面の末尾に折りたたみ**で置く（新画面・新ナビ項目は作らない）。Todayの「完了済み n件 ▸」と同じパターン。

```
Inbox                     未仕分け n件
[未仕分けリスト] 各行に「今日」「明日」
＋ 入力欄（Smart Input）
▸ この先の予定 n件
```

Todayの末尾ではなくInboxに置く理由: 「明日」で仕分けたタスクが**同じ画面の上のセクションから下のセクションへ移動**するだけになり、消えた感覚が生まれない。Todayは「今日だけ」の純度を保てる。

### 12.3 中身

- 対象: `kind='todo' AND status='todo' AND due_date > 今日`。**プロジェクトの子タスクも含む**（親の有無を問わず「この先の予定」を全て見る）。is_memoは除外しない（Todayと同じ扱い＝期日付きメモは予定として出る）
- 並び: `due_date` 昇順
- 行はTodayと同じ部品を流用: チェック円（その場で完了可）／タイトル＋メタ行／タップで詳細モーダル
- **「明日へ」ボタンは出さない**（未来のタスクに対して意味が薄い）。**「今日へ」引き上げボタンも作らない**——日付変更は詳細モーダルの期日ピッカーで行う（行を軽く保つ）
- 件数バッジ（ナビのInbox）は従来通り**未仕分けのみ**を数える（予定は含めない）

### 12.4 繰り返しの棚卸し

メタ行に「**繰り返し**」チップ（asagi）を追加する。設計4.2により繰り返しタスクは常に「次の1件」だけ存在するため、**この一覧がそのまま繰り返し設定の棚卸しリストになる**（専用画面は不要）。チップはTodayにも同じく表示する（共通部品 `TaskMeta`）。

### 12.5 API

既存 `GET /api/items` に `due_after=YYYY-MM-DD`（`.gt("due_date", ...)`）を追加するのみ。`due_before` の対で、新エンドポイントは作らない。

## 15. リマインダー配信（実装済み）

「忘却をシステムで防ぐ」というアプリの中核（spec.md 1.2）。ルール→`remind_at` の解決は既存（6.1）で、本章は**配信層とその周辺UI**。

### 15.1 全体構成

```
外部cron --毎分--> GET /api/cron/reminders (Bearer CRON_SECRET)
  1. 発火対象を取得: sent_at is null かつ (snoozed_until ?? remind_at) <= now
  2. 先に sent_at をマークして所有権を取る（at-most-once）
  3. 親アイテムを引いて仕分け（15.4）
  4. push_subscriptions 全件へ web-push 送信
```

- Vercel Hobbyのcronは分単位で回せないため**外部cronサービス**（cron-job.org 等）を使う。Supabase pg_cron はベンダー結合になるため不採用（database-design.md 6.2）
- 2の所有権取得は `.update(...).in("id", ids).is("sent_at", null).select()`。**取れた行だけが自分の担当**になるので、cronが多重起動しても二重送信しない
- 送信結果は `{ checked, sent, discarded, delivered, subscriptions }` で返す（cronサービスのログで状況が読める）

### 15.2 通知の中身

| 要素 | 値 |
|---|---|
| タイトル | タスク名 |
| 本文 | 「7月22日（水） 16:00 ・ 1時間前」（期日＋ルール表記。既存の `formatDueFull` / `formatReminderRule` を再利用） |
| tag | `item.id`。**同じタスクの通知は上書き**して通知欄を溢れさせない（`renotify: true` で再通知時もちゃんと鳴る） |
| アクション | **「完了」1つだけ**。SWから `POST /api/items/:id/complete` を叩く＝アプリを開かずに完了できる |
| タップ | `/today?item=<id>` を開き、**そのタスクの詳細モーダルが開いた状態**で着地する |

アイコン（`icon`=`/icons/icon-192.png` / `badge`=`/icons/badge-72.png`）は16章のPWAアイコン生成で用意し、`sw.js` の `showNotification` に固定で埋めた。

### 15.3 通知タップの着地

`/today?item=<id>` は `searchParams` をサーバー側で受けて `TodayView` に渡す（`useSearchParams` はSuspense境界を要求し、本番ビルドが落ちるため使わない）。モーダルを閉じたら `replaceState` でクエリを消す（リロードで再び開かないように）。SWは既存タブがあれば `navigate` + `focus` し、タブを増やさない。

### 15.4 送らない通知の判定（`src/lib/push-dispatch.ts`）

**完了・破棄したタスクの通知が後から鳴る問題**を、ディスパッチャ側で一元的に塞ぐ。`complete` / `drop` などのルートはリマインダーを触らない（4ルートに同じ後始末を書かない）。

| 条件 | 扱い |
|---|---|
| 親アイテムが `status != 'todo'` | 送らない（`sent_at` は埋めるので再評価もしない） |
| 親アイテムが存在しない | 送らない |
| 発火予定が**24時間以上前** | 送らない。cron停止からの復帰時に古い通知が一気に鳴るのを防ぐ。タスク自体はTodayに残るので取りこぼしにはならない |

### 15.5 通知許可のフロー

**起動時にプロンプトを出さない**（一度ブロックされるとアプリ側からは二度と戻せないため）。導線は2つ:

1. **Settings画面の「通知」セクション** — 状態表示＋「通知を許可」/「止める」＋「テスト通知を送る」
2. **詳細モーダルのリマインダー行の直下** — リマインダーが1件以上あるのに通知が届かない状態のときだけ、1行の案内＋「通知を許可」（`PushNotice`）。**通知を望んだ瞬間に聞く**のが狙い

状態は `Notification.permission` と `pushManager.getSubscription()` の両方から導出（`src/lib/push-client.ts` の `PushState`）:

| 状態 | 表示 |
|---|---|
| `unsupported` | 「このブラウザは通知に対応していません」（操作なし） |
| `default` / `granted` | 「この端末では通知が届きません」＋許可ボタン |
| `denied` | 「ブラウザで通知がブロックされています」＋サイト設定への案内（ボタンは出さない） |
| `subscribed` | 「この端末で通知を受け取ります」＋止めるボタン＋テスト通知 |

購読は端末ごとに1行（`endpoint` でupsert）。送信失敗時は 404/410 なら即削除、それ以外は `failed_count` を数えて5回で削除する。

### 15.6 スヌーズ（UIは保留）

spec.md 10.1 はMVPにスヌーズを含むが、通知アクションを「完了」1つに絞ったため置き場所が未定になった。**ディスパッチャは `snoozed_until` を優先して発火する実装になっている**ので、UIを足すだけで後付けできる。実運用してから必要性を判断する。

### 15.7 検証済みの挙動

ユニットテスト12件（24時間ルール・完了済み除外・スヌーズ優先・ペイロード生成）＋API E2E 20項目（認証401/200・二重送信しない・`sent_at` の埋まり方・購読のupsert/解除・送信失敗時の `failed_count`）。ブラウザではSW登録（`/sw.js` が `no-cache` で配信される）・許可状態3分岐の表示・`?item=` からのモーダル自動オープン・モバイル/ダークモードを確認。

## 16. PWA化（インストール可能まで・実装済み）

## 17. 体感速度の改善（SWR＋楽観的更新＋API並列化・実装済み）

運用開始後、体感レスポンスの悪さが最大のストレスになったため対処した。指摘は2つ:「タスク登録がTodayに出るまで一拍ある」「タブ切り替えで一覧表示が遅い」。

### 17.1 原因（3つ）

1. **登録が楽観的でない**: 追加が `await postJson` の応答を待ってから一覧に反映していた
2. **タブ切り替えで毎回ゼロから再取得**: 全5ビューが `useState+load()+useEffect` で、ナビのたびに再フェッチ＋「読み込み中…」を挟んでいた（クライアントキャッシュなし）
3. **/api/today が5クエリ直列**: todos→done→habits→logs→instances を順に `await`。独立クエリなのに往復が積み上がっていた

前提として **Vercelのリージョンを Supabase に揃える**（1往復の短縮）を先に実施済み。リージョン不一致だと全クエリが太平洋を往復して桁違いに遅くなる（環境の落とし穴に記載）。

### 17.2 対処

| 対象 | 内容 |
|---|---|
| キャッシュ | **SWR導入**（`src/components/swr-provider.tsx` の `SWRConfig`）。`fetcher`=既存 `getJson`、`revalidateOnFocus`（二拠点の鮮度）、`keepPreviousData`、`dedupingInterval:2000`。グローバルキャッシュはアンマウント後も残るので、タブを戻ると**キャッシュ即描画→裏で再取得**になる |
| 全ビュー | today/inbox/notes/projects/habits を `useSWR(KEY)` に統一。「読み込み中…」は初回（`isLoading && !data`）のみ |
| 楽観的更新 | 追加は一時ID（`temp-*`・`makeOptimisticItem`）で即表示→応答で本物に置換。完了/先送り/仕分け/アーカイブ等は `mutate` の `optimisticData`＋`rollbackOnError` で即時反映し、失敗時は自動巻き戻し |
| 横断無効化 | `INBOX_CHANGED_EVENT`/`notifyInboxChanged` を廃止し `mutate` に統一。**Inboxバッジと一覧は同一 `INBOX_QUERY` キーを共有**＝1フェッチに集約され、仕分け1回で両方追従。詳細モーダル閉時は `revalidateLists()`（`/api` 全キー再検証）で横断更新 |
| API並列化 | `/api/today` の5クエリを `Promise.all` に（往復5回→1回相当）。計算ロジックは不変 |

### 17.3 判断メモ

- 楽観更新の仮アイテムは `id` が `temp-` 接頭辞。仮の間はチェック/タップを無効化し、確定後の本物に差し替わる
- 習慣のTodayタスク化・追加は `/api/habits` と `/api/today` の両方に波及するため、処理後に両キーを `mutate`
- **SWRのメモリキャッシュはオフライン対応ではない**（リロード/機内では消える）。将来オフライン化する際は SWRの `provider` を IndexedDB 永続キャッシュに差し替える発展余地がある（14章と接続）

### 17.4 検証済みの挙動

型チェック・テスト95件維持。ブラウザ（dev）で: **Today/Inboxの追加が50ms以内に即表示・応答後も重複なし**／**タブを戻ると「読み込み中…」を挟まずキャッシュ即描画**／Inbox追加でバッジが即 0→1（同一キー共有）／`/api/today` は単一リクエスト／PC・モバイル・ダークモードで回帰・エラーなし。
